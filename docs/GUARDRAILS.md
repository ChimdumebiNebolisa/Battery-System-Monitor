## Workflow Guardrails

### Review context before planning or coding
Before proposing a plan or writing code, deeply review the relevant context, files, existing patterns, dependencies, constraints, and documentation.

Make sure you understand how the current system works first.
Do not make shallow assumptions.
Base decisions on what you actually find in the project, not what you expect to be there.

If something is unclear, inspect more before acting.
Do not invent architecture, file relationships, or conventions without evidence.

### Keep the README current
Update the README whenever the project changes in a way that affects:
- what the project does
- how to build it
- how to run it
- how to understand its structure
- what scenarios or features are supported

The README should stay aligned with the actual project state.
Do not let it become stale, aspirational, or disconnected from the codebase.

If setup steps, commands, project structure, or behavior change, reflect that in the README.
